01 — Project Overview

Complete restaurant management without internet dependency

This system solves a critical problem in restaurant operations: internet dependency. Most modern POS and restaurant management solutions fail when the network goes down. This ecosystem runs entirely offline with a local backend server, real-time communication over LAN, and optional cloud sync when available.

Built as a complete multi-app system covering host management, POS billing, kitchen operations, customer feedback, table devices, and token displays. All 10 applications communicate in real-time without external servers.

02 — Applications in the Eco-System

10 interconnected applications

Host App

Runs the local backend server (Dart Shelf). Manages table assignments, waiting lists, and acts as the central hub for all other applications.

POS (Point of Sale)

Complete billing system with multi-payment support, split bills, discounts, tax calculation, and invoice generation. Supports USB, Bluetooth, and LAN thermal printers.

KDS (Kitchen Display System)

Real-time order display for kitchen staff. Multi-kitchen support with order routing, status updates, and prep time tracking.

Captain App

Waiter/server interface for tableside ordering. Orders sync directly to KDS and POS without manual entry.

Token Display

Large-screen display showing order tokens and statuses. Customers see live updates when orders are ready for pickup.

Waiting App

Queue management for walk-in customers. Displays estimated wait times and notifies when tables are available.

Table Device Configuration

Configuration tool for IoT table devices with three-button interface: call waiter, request water, and request bill. Customers use these devices for instant service requests without leaving their table.

KOT Auto Print

Kitchen Order Ticket (KOT) auto-printing service. Routes orders to designated kitchen printers based on item category.

Feedback App

Customer feedback collection interface. Captures ratings and comments immediately after dining.

Customer Display

Secondary display at POS counter showing order details to customers during checkout.

04 — System Architecture

How 10 applications communicate offline

The Host App runs a Dart Shelf server that acts as the backend for all other applications. When any app launches, it discovers the Host App on the local network and establishes a WebSocket connection for real-time updates.

All business logic lives in the local server — order management, billing, table assignments, kitchen routing. Applications communicate via REST APIs for actions and WebSocket for live state synchronization. This ensures instant updates across all devices when an order is placed, modified, or completed.

Data persistence uses Isar database for lightweight local storage and PostgreSQL for production deployments. An optional Node.js backend handles cloud sync when internet is available — pushing daily sales, analytics, and backups without interrupting live service.

05 — Hardware & Device Integration

Connecting physical devices to software

Thermal printers are integrated via USB, Bluetooth, and LAN. The system auto-detects connected printers and supports ESC/POS commands for invoice and KOT printing. Multiple printers can be configured per kitchen or billing counter.

Each table is equipped with a small service device that lets customers send service requests with a single button press. When a button is pressed, the request is instantly shown to restaurant staff on their screens, helping them respond faster and improve the overall dining experience.

The system is used across different screens inside the restaurant, including billing counters, kitchen screens, service staff devices, and customer-facing displays. Each screen runs a dedicated application designed for its specific role, ensuring smooth coordination between staff and a better experience for customers.

09 — Why This Project Matters

Real-world impact and learnings

This project demonstrates the ability to design and build complex, production-grade systems that solve real business problems. It required deep understanding of system architecture, network communication, hardware integration, and offline-first data management.

The system is deployed in live restaurant environments, handling hundreds of daily orders without internet dependency. It proves that Flutter can be used beyond mobile apps — for backend servers, IoT communication, and multi-platform desktop applications.

Key takeaway: offline-first architecture isn't just a feature, it's a reliability guarantee. When designed correctly, local systems perform better, faster, and more reliably than cloud-dependent alternatives.
